# Pull OTA — Self-Updating Firmware for the Coati Clock

**Date:** 2026-04-06  
**Status:** Implemented (basic HTTP); authenticated HTTPS flagged as future need

## Why

ESPHome's built-in OTA is **push**: the developer's machine connects inbound to port 3232 on the device. This works fine on the local LAN but fails when a device is behind a NAT'd router or firewall where inbound connections are blocked.

Pull OTA inverts this: the device periodically fetches a firmware manifest from an HTTP server it can reach, compares versions, and self-updates if a newer build is available. The device only initiates outbound connections — no inbound ports required.

The **Coati Clock** is a project intended for multiple deployed devices. `bb32` is the hostname of the first/reference unit; future units (`bb33`, etc.) will run the same codebase with device-specific configuration. The pull OTA system is designed with this fleet in mind.

## Architecture

```
Developer machine
  └─ esphome compile bb32.yaml          (builds bb32-specific firmware.ota.bin)
  └─ ./deploy.sh bb32                   (scp to rpi4, updates bb32 manifest)
  └─ ./deploy.sh bb33                   (same for next device)

rpi4 (192.168.153.78)
  └─ ~/firmware/                        (served on port 8080)
       ├── bb32_manifest.json           (version + path + MD5 for bb32)
       ├── bb32.ota.bin
       ├── bb33_manifest.json           (version + path + MD5 for bb33)
       └── bb33.ota.bin

Coati Clock devices
  └─ bb32 (192.168.69.164) polls bb32_manifest.json every 6h
  └─ bb33 (next unit)      polls bb33_manifest.json every 6h
  └─ each: if version > running → downloads its own .ota.bin → self-flashes → reboots
```

### Why each device gets its own binary

ESPHome compiles device-specific secrets (hostname, Stra2us client ID, HMAC secret, OTA password) as string constants directly into the firmware binary. Two Coati Clock units running the same codebase will produce **different binaries** because they have different credentials baked in. A shared firmware binary is not possible without significant changes (e.g. storing credentials in NVS at provisioning time, outside the build system).

This means:
- One manifest + one `.ota.bin` **per device**
- `deploy.sh` takes the device name as an argument
- The push OTA and pull OTA workflows are both per-device

The push OTA path (`esphome run`) remains available for local development — both coexist in each device's YAML.

## Components

### bb32.yaml

Three blocks work together:

```yaml
http_request:            # HTTP client used by update and ota components
  timeout: 30s
  useragent: esphome/bb32

ota:
  - platform: http_request   # pull OTA downloader + flasher
    id: ota_http
  - platform: esphome        # push OTA (local dev, unchanged)
    password: "..."

update:
  - platform: http_request   # manifest poller + version comparator
    name: "bb32 Firmware"
    source: "http://${firmware_host}:8080/bb32_manifest.json"
    update_interval: 6h      # checks at boot, then every 6h
```

`firmware_host` is defined in `secrets.yaml` so the rpi4 IP stays out of version control.

### Manifest format

The manifest is a JSON file read by the device on each update check:

```json
{
  "name": "bb32",
  "version": "2026.04.06.1630",
  "builds": [
    {
      "chipFamily": "ESP32-C3",
      "ota": {
        "path": "http://192.168.153.78:8080/bb32.ota.bin",
        "md5": "a1b2c3d4...",
        "summary": "Built 2026.04.06.1630"
      }
    }
  ]
}
```

- `version` is compared against the **project version baked into the running firmware** (see [Build Versioning](#build-versioning) below). If the manifest version is lexicographically greater, the update is applied.
- `md5` is verified by the device after download to detect corruption.
- `chipFamily` must match `"ESP32-C3"` exactly — other chip entries are ignored.

The version string generated by `deploy.sh` is `YYYY.MM.DD.HHMM` (e.g. `2026.04.06.1630`), which sorts correctly as a plain string comparison.

### Build Versioning

The build version is a `YYYY.MM.DD.HHMM` datecode (e.g. `2026.04.06.1630`) injected into the binary at compile time. It appears in three places that must all match:

| Location | How it gets there |
|---|---|
| Running firmware | `ESPHOME_PROJECT_VERSION` macro, set via `!env_var ESPHOME_PROJECT_VERSION` in `bb32.yaml` |
| Manifest `version` field | Written by `deploy.sh` with the same timestamp |
| Uptime report (`fw=...`) | `ESPHOME_PROJECT_VERSION` in the `snprintf` format string |

`deploy.sh` generates the version *before* calling `esphome compile` and exports it as `ESPHOME_PROJECT_VERSION`. ESPHome's `!env_var` tag picks it up automatically:

```bash
VERSION="$(date '+%Y.%m.%d.%H%M')"
export ESPHOME_PROJECT_VERSION="$VERSION"
python3 -m esphome compile bb32.yaml
```

A **local build** (running `esphome compile` directly without `deploy.sh`) uses the default value `dev`:

```
up=0d0h5m heap=184320 heap_lo=160240 rssi=-61 rst=SW fw=dev
```

A **deployed build** shows the datecode:

```
up=0d6h42m heap=184320 heap_lo=160112 rssi=-61 rst=SW fw=2026.04.06.1711
```

The `fw=` field in the uptime report is the single authoritative confirmation that a pull OTA applied successfully.

### deploy.sh

```bash
./deploy.sh bb32           # compile bb32 → scp bin + manifest to rpi4 → device updates within 6h
./deploy.sh bb32 --flash   # same, plus push OTA to device immediately (requires LAN access)
```

The device name argument selects which `<name>.yaml` to compile and which manifest to update on rpi4. Internally:
1. Generates version datecode `YYYY.MM.DD.HHMM` and exports as `ESPHOME_PROJECT_VERSION`
2. Activates `esphome_2026` virtualenv
3. Runs `esphome compile <name>.yaml` — the version is baked into the binary via `!env_var`
4. Computes MD5 of `firmware.ota.bin`
5. Writes manifest with the same version string
6. `scp`s both files to `rpi4:~/firmware/<name>.ota.bin` and `<name>_manifest.json`

## One-time rpi4 Setup

```bash
# On rpi4
mkdir -p ~/firmware
cd ~/firmware

# Start HTTP server — must run from ~/firmware so files are served correctly
python3 -m http.server 8080 --bind 192.168.153.78
```

Binding to a specific IP (`--bind 192.168.153.78`) limits exposure to that interface only, rather than accepting connections on all interfaces.

To persist across reboots, create a systemd unit:

```ini
# /etc/systemd/system/firmware-server.service
[Unit]
Description=Firmware HTTP server
After=network.target

[Service]
User=austin
WorkingDirectory=/home/austin/firmware
ExecStart=/usr/bin/python3 -m http.server 8080 --bind 192.168.153.78
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable --now firmware-server
```

## Security Posture

### What's in the .bin

The compiled firmware binary contains secrets as readable strings:

| Secret | Extractable with `strings`? |
|---|---|
| Stra2us HMAC secret hex | **Yes — plaintext** |
| Stra2us client ID | **Yes — plaintext** |
| firmware_host IP | **Yes — plaintext** |
| WiFi SSID / password | No (binary struct form) |
| OTA password | No (hashed) |

**The `.bin` and `.ota.bin` files must be treated as sensitive.** Do not commit them to git (they are in `.gitignore`), do not serve them on a public-facing port, and do not share them.

### Current mitigations

- HTTP server binds to a single LAN interface (`--bind 192.168.153.78`), not `0.0.0.0`
- `firmware_host` and Stra2us credentials are in `secrets.yaml`, which is gitignored
- The firmware directory is not browsable outside the local subnet

### Known limitation

The firmware HTTP server has no authentication. Any host on the `192.168.153.x` subnet can download `bb32.ota.bin` and extract the HMAC secret using `strings`.

---

> [!WARNING]
> **Future need: authenticated HTTPS firmware server**
>
> The current unauthenticated HTTP server is acceptable for a trusted home LAN but should be hardened before any deployment in a less-controlled environment.
>
> **Recommended upgrade:**
> 1. Add TLS to the firmware server (self-signed cert is fine for a private LAN, or use a Let's Encrypt cert if rpi4 has a public hostname)
> 2. Add HTTP Basic Authentication to the firmware endpoint so only authorised clients can fetch the binary
> 3. Update `bb32.yaml` to use `https://` in the manifest URL and add credentials:
>    ```yaml
>    http_request:
>      timeout: 30s
>      useragent: esphome/bb32
>      # Basic auth for firmware server
>      headers:
>        Authorization: "Basic <base64(user:password)>"
>    ```
>    The Basic Auth header would be added to `secrets.yaml` (base64-encoded, still in the binary — but the server rejects anyone without it, meaningfully raising the bar).
>
> This eliminates casual LAN extraction of the binary while keeping the deployment simple (no PKI, no device certificates required).

## Future: On-demand Trigger via Stra2us (Option 2)

The 6h polling interval handles the "device wakes up after a long sleep and pulls the latest" case automatically. For on-demand triggering without waiting up to 6h, a Stra2us command can force an immediate check:

```yaml
# In bb32.yaml — on_message handler for a command topic
- platform: stra2us
  topic: bb32/cmd
  on_message:
    - if:
        condition:
          lambda: 'return x == "ota_check";'
        then:
          - lambda: 'id(firmware_update).perform();'
```

This requires no new firmware download path — `perform()` triggers the existing `update:` component immediately. The combination of hourly/daily polling (Option 1) and on-demand trigger (Option 2) gives the full Particle-style experience:

- **Sleeping/offline device:** pulls latest on next wake via polling ✅
- **Online device:** operator can push "check now" command via Stra2us ✅
- **No inbound connections required:** both paths are device-initiated ✅
